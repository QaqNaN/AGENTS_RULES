# AGENTS.md

## 基本原则

1. 默认采用最小必要改动。
   只修改与当前需求直接相关的代码，不扩大范围，不顺手重构，不修改无关文件。

2. 最小改动不等于临时补丁。
   应优先选择最直接、局部、清晰的修改方式。  
   当简单的内联修改符合现有代码风格时，不要引入多余的间接层、临时变量、包装函数、适配层或推测性的扩展设计。

3. 优先复用现有接口、方法、工具函数、组件和代码风格。
   在新增接口、方法、配置项、依赖或抽象层之前，必须先检查现有实现是否已经能满足需求。

4. 禁止未经允许的大规模重构。
   不要主动调整架构、目录结构、命名体系、模块边界、状态管理方式、数据模型、公共 API 或 UI 设计语言。

5. 保持现有行为。
   除非用户明确要求改变，否则不要改变已有业务逻辑、外部行为、配置格式、接口返回值或数据结构。

## 工作流程

6. 当用户没有明确要求实现时，不要修改代码。
   如果用户只是提出想法、询问可行性、要求分析、调试建议、设计评审或方案对比，应先给出最多三种方案。  
   每种方案需要简要说明：
   - 优点
   - 缺点
   - 负面影响
   - 最小可行改动

7. 只有在用户明确表达“实现”“修改代码”“帮我改”“修复”“编辑”“应用方案”等意图后，才可以进入代码修改阶段。

8. 当需求不清晰时，不要直接写代码。
   应先澄清目标行为、输入输出、边界条件、异常处理方式和影响范围。

9. 对于小而明确的改动：
   - 直接进行相关代码修改
   - 运行最小相关检查
   - 总结修改内容和验证结果

10. 对于复杂、含糊或涉及多文件的改动：
    - 先阅读相关文件
    - 修改前给出简短实现计划
    - 优先增量修改，避免大范围重写
    - 不修改无关代码
    - 不引入不必要的新模式

## 阅读和修改代码

11. 修改文件前，应阅读足够的上下文。
    包括附近代码、调用方、相关工具函数、已有组件和本地代码风格。

12. 修改公共函数、导出组件、共享工具或公共接口前，应先检查相关调用点。

13. 修改前先判断影响范围。
    应优先定位最小修改点，避免跨模块扩散。  
    如果必须修改多个文件，应说明每个文件的修改原因。

14. 不要保留本次变更后已经明确废弃的代码。
    如果某些变量、代码块、方法、分支逻辑或旧流程已经被本次变更明确替代，应一并删除。  
    但不要清理与当前需求无关的历史代码。

15. 不要引入没有实际需求的扩展性设计。
    禁止为了“未来可能需要”而新增抽象层、配置项、策略类、回调、插件机制或复杂数据结构。

16. 保持现有代码风格。
    命名、类型标注、日志、异常、目录结构、注释风格应与当前项目一致。  
    除非现有风格明显导致错误，否则不要主动改风格。

## 调试规则

17. 调试时应优先尝试复现问题。

18. 修复根因，不要只修复表面症状。

19. 禁止为了让问题消失而删除测试、削弱类型检查、禁用 lint 规则、吞掉异常或改变业务语义。

20. 如果存在多个合理修复方案，应选择最符合当前代码结构、侵入性最低的方案，并简要说明取舍。

## 错误处理和日志

21. 禁止静默处理错误。
    不允许吞异常、不允许空 catch、不允许失败后假装成功、不允许用默认值掩盖错误。

22. 对于未预期的状态，应直接抛出异常，让问题尽早暴露。
    不要在没有用户明确允许的情况下加入自动降级、自动重试、默认 fallback 或忽略逻辑。

23. 应用错误和未预期异常必须通过项目现有日志系统记录。
    不要使用 `print`、`console.log` 或临时调试输出处理错误，除非项目当前约定就是这样做的。

24. 正常的 CLI 输出或用户可见输出，可以使用项目已有的输出方式。

25. 只有在已有明确业务语义时，才可以捕获异常。
    捕获异常后必须进行有意义的处理，例如补充上下文后重新抛出、转换为更明确的业务异常，或执行用户明确要求的恢复逻辑。

26. 不得在日志中输出密钥、token、密码、凭证或敏感数据。

## 测试和验证

27. 修改完成前，应运行能证明改动有效的最小相关检查。

28. 前端改动优先运行：
    - 类型检查
    - lint
    - 相关测试
    - 当用户流程变化时，再运行端到端测试

29. 后端改动优先运行：
    - 相关单元测试或集成测试
    - 类型检查，如果项目适用
    - 数据库相关改动需要仔细验证查询、迁移或 schema 变化

30. 不要默认运行耗时很长的全量测试，除非用户要求或改动影响范围较大。

31. 如果无法运行检查，应在最终回复中明确说明原因。

32. 禁止伪造验证结果。
    只有实际运行过的测试、类型检查、lint、构建或手动验证，才可以声明已通过。

33. 如果检查失败，应直接报告失败信息。
    不要隐藏、忽略或绕过失败，除非用户明确要求临时处理。

## 文件、编码和命令

34. 始终使用 UTF-8 读写文本文件。

35. 设置控制台输出编码为 UTF-8。

36. 运行 PowerShell 命令时，优先使用 `login:false`，避免加载用户 profile 脚本，除非确实需要。

37. 不要修改生成文件，除非当前任务明确要求。

38. 不要修改 lockfile，除非依赖变化是用户明确要求且确实必要的。

## 依赖和配置

39. 使用项目当前已有的包管理器。
    不要切换包管理器。

40. 不要新增生产依赖，除非用户明确允许。
    新增依赖前，应先检查项目中是否已有等价工具或实现。

41. 不要更新无关依赖。

42. 不要硬编码密钥、token、API key、密码、凭证、私有 URL 或机器特定路径。

43. 不要修改 `.env`、`.env.local` 或生产配置文件，除非用户明确要求。

44. 如果确实需要新的配置项，应说明变量名、用途和期望格式。

## 保护已有改动

45. 修改前应尽量检查当前 diff 或文件状态。

46. 不要覆盖与当前任务无关的用户已有改动。

47. 如果文件中已经存在无关修改，应保留这些修改。

48. 如果当前需求与已有未提交改动冲突，应停止并询问用户。

49. 禁止运行破坏性 git 命令，例如 `reset`、`checkout`、`clean`、`rebase`，除非用户明确指示。

## 需要先询问的操作

执行以下操作前必须先询问用户：

- 大规模重构
- 数据库 schema 或 migration 变更
- 认证、授权、支付或安全相关变更
- 新增生产依赖
- 修改公共 API
- 修改配置格式
- 修改数据模型
- 删除或重命名大量文件
- 批量格式化、批量重命名或大范围移动代码

## 禁止行为

禁止：

- 在没有明确要求时修改代码
- 修改无关代码
- 为了通过测试而改变业务语义
- 删除测试来让测试通过
- 忽略类型错误、禁用 lint 规则或吞掉异常
- 用复杂抽象掩盖简单问题
- 用临时 workaround 替代直接修复，除非用户明确要求
- 添加未请求的功能
- 重写无关模块
- 静默删除用户代码
- 在没有说明的情况下改变外部行为

## 最终回复

当代码被修改后，最终回复必须包含：

- 修改了哪些文件
- 改了什么行为
- 是否删除了旧逻辑
- 运行了哪些检查，以及结果
- 哪些检查没有运行，以及原因
- 是否存在兼容性影响、风险、假设或后续工作

任务只有在满足以下条件时才算完成：

- 用户请求的行为已经实现
- 相关检查已通过，或无法运行的原因已说明
- 没有修改无关文件
- 错误被修复，而不是被隐藏
- 风险、假设和后续工作已经明确说明
